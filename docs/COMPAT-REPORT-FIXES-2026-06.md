# Compatibility-report fixes — v1.0.4 (2026-06)

> Lineage note: live Nexus/Steam was at **v1.0.3** when this batch was cut, so
> this release is **v1.0.4**. The repo's SubModule.xml and git tags had drifted
> (stuck at v1.0.0) because 1.0.1–1.0.3 were shipped via the build `-Version`
> param without bumping the source file or tagging — fixed going forward.

Three post-v1.0.0 Nexus user reports, each tracing to a distinct code path. All
fixed in one batch; mapping below is the authoritative record of what each slice
addresses and why.

## Report → root cause → fix

### R1 — Foreign-MCM mods don't register / panels UI-blocked (Artem suite)
> "some mods aren't being registered at all, or some are completely UI blocked …
> especially offsite ones like the Artem suite."

Mods compiled against the **real** BUTR MCM assembly register their settings
through it; those instances reach our registry wrapped in
`SettingsRegistry.ForeignSettingsAdapter`. The old `SettingsVM.BuildGroups`
reflected over the adapter *shell* (zero `[SettingProperty*]` members) → rendered
a **blank panel** and logged a one-shot CompatWarn instead of the settings.

**Fix (S1):** `SettingsVM` reflects over `adapter.Wrapped`; `SettingsPropertyVM`
`LiveGet`/`WriteBack` detect `ForeignSettingsAdapter` on the owner and read/write
the wrapped instance. Foreign-MCM mods now render and persist.
Files: `SettingsVM.cs`, `SettingsPropertyVM.cs`.

### R2 — Group items locked with no enable toggle (Artem Cinematic "Run")
> "Artem cinematic locks some items under 'Run' without a toggle to disable."

`[SettingPropertyGroup(IsMainToggle=true)]` declares a per-group master switch.
The builder folded it into the child-property list and never surfaced it, so the
group's items had no enable/disable control.

**Fix (S2):** the main-toggle property is pulled out and rendered as a group-header
checkbox (`HasGroupToggle`/`GroupToggleValue`/`ExecuteToggleGroup`) that shows or
hides every child row. Toggle-only groups still appear.
Files: `SettingsPropertyGroupVM.cs`, `SettingsVM.cs`.

### R3 — No way to toggle ButterLib subsystems from the UI
> "the native JSONs don't incorporate txt disables on sub systems."

ButterLib subsystems (`ISubSystem`) had no settings surface and no persistence —
nothing for the user to toggle, and any state wouldn't survive a restart.

**Fix (S3):** synthetic "Sub Systems" MCM page. `SubSystemBridge` (in Foundation)
decouples MCM↔ButterLib; ButterLib wires the roster at load; `SubSystemSettingsPage`
is auto-registered when the bridge is live; `subsystems.json` persists state;
`SubSystemManager` defers-disables saved-off subsystems so they never start.
Files: `SubSystemBridge.cs`, `SubSystemSettingsPage.cs`, `SubSystemPersistence.cs`,
`SubSystemManager.cs`, `BetaDepsButterLibSubModule.cs`, `SettingsRegistry.cs`,
`SettingsVM.cs`, `SettingsPropertyVM.cs`.

### R4 — UIExtenderEx mods lose features on 1.2+ (Party AI Controls, Clan Control)
> "mods that depend on UIExtenderEx (Party AI Controls, Clan Control etc) … would
> have some of their features disable or not working."

BL 1.2+ ships **code-generated** prefabs (`*.GauntletUI.AutoGenerated*.dll`).
`GauntletMovie.Load` instantiates those via `GeneratedPrefabContext` and never
calls `WidgetPrefab.LoadFrom(path)` — the hook our prefab patcher relies on. So a
`[PrefabExtension]` targeting a generated prefab (Party screen, Clan screen, etc.)
silently never applied: the mod's ViewModel mixin attached but its injected
button/panel was missing → "feature disabled / not working."

**Fix (S4):** `GeneratedPrefabGate` Harmony-patches `GauntletMovie.Load` to force
`doNotUseGeneratedPrefabs = true` when any prefab patch is registered, routing the
movie (and its nested sub-prefabs) down the XML path where the hook injects. Same
approach upstream UIExtenderEx takes. Cost = the per-movie XML parse the generated
prefabs optimized away — how every UIExtenderEx-modded install has always run.
Files: `GeneratedPrefabGate.cs`, `BetaDepsUIExtenderExSubModule.cs`.

## Bonus — MCM localization
`{=key}` display strings now resolve through `TaleWorlds.Localization.TextObject`
first (guarded; falls back to the static strip), so non-English language packs
apply. Was always showing the English fallback. Addresses the separate
"localization doesn't work (MCM)" report. File: `TextHelper.cs`.

## Build / platform
Foundation, Harmony, UIExtenderEx, ButterLib, MCM now multi-target
`net472;net6.0` (Game Pass / net6 runtime). MCM `AssemblyVersion` stays
`5.11.99.0` (impersonates upstream BUTR MCMv5 — must not change).
